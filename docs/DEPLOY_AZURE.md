# Деплой Staffing Agent вне ноутбука (в т.ч. Azure)

Короткий чеклист: что переносится «как есть», что нужно **явно перевести с локальных привычек** (профили, файлы, интерактивный логин).

## 1. Как устроен бот сейчас

- **Slack Socket Mode** — один **долгоживущий** процесс: `python -m staffing_agent`.
- Нужен хост, где процесс **не усыпляют** (не scale-to-zero без оговорок), и стабильный **исходящий** интернет к Slack / API.

## 2. Шаги деплоя (универсально)

1. **Сборка окружения** — Python 3.11+, `pip install -r requirements.txt`, запуск из корня репо (рядом с `staffing_agent/`).
2. **Секреты** — все значения из `.env.example` задать **в секретах платформы** (не коммитить `.env`, не печатать в логи). Минимум Slack: `SLACK_BOT_TOKEN`, `SLACK_SIGNING_SECRET`, `SLACK_APP_TOKEN`.
3. **Проверка перед боем** — один раз: `python -m staffing_agent --check` (и при необходимости `--check-llm`).
4. **Автозапуск** — systemd / policy рестарта контейнера / min replicas = 1 в оркестраторе.
5. **Логи** — stderr процесса (и при необходимости файловый лог из кода), алерт если процесс упал.

## 3. Где обычно «переезжают с локального»

| Область | Локально часто так | В инфре нужно |
|--------|---------------------|---------------|
| **Slack** | Токены в `.env` | То же; токены в Key Vault / секретах сервиса. |
| **Databricks** | `DATABRICKS_PROFILE` + `databricks auth login` (интерактив), креды в `~/.databricks` | **Явный неинтерактивный вход**: PAT, Azure AD service principal + secret (или OIDC из пайплайна), переменные окружения/host/workspace id, как ожидает ваш `databricks` CLI/SDK. Профиль «с ноутбука» сам по себе на сервер не переедет — нужен секрет и доки по auth для **Azure Databricks**. |
| **Notion** | `NOTION_API_KEY` в `.env` | То же из секретов; исходящий HTTPS. |
| **Anthropic / LiteLLM** | Ключ или base URL в `.env` | То же из секретов; **исходящий HTTPS** с подсети Azure. Cato на ноутбуке ≠ сеть воркера — см. **§5**. |
| **Google Docs (опционально)** | JSON key на диске, путь в env | Secret как файл/volume или base64 + запись при старте; не в git. |
| **CSV из Notion** | Файлы под `config/notion_export/` | Либо кладут в образ/диск при деплое, либо обновляют пайплайном; путь через `STAFFING_PEOPLE_CSV_PATH` и т.д. |

**Главная миграционная точка для вас** — **Databricks**: перестать опираться на интерактивный логин и локальный профиль; зафиксировать **какой именно механизм** (PAT vs SP/OIDC) допускает ваша Azure-политика.

## 4. Варианты внутри корпоративной Azure

Все ниже — «always-on воркер под Socket Mode», плюс Key Vault для секретов.

| Вариант | Плюсы | На что смотреть |
|---------|--------|-----------------|
| **Azure VM** (+ cloud-init / Ansible) | Просто отладить, как на сервере; полный контроль | Патчинг ОС, сам systemd. |
| **Azure Container Apps** | Контейнер, секреты, **min replicas = 1** чтобы не рубить WebSocket | Убедиться, что политика масштабирования не опускает реплики до 0. |
| **AKS (один Deployment)** | Если у вас уже кластер | Overkill для одного бота, если нет стандарта. |
| **Azure Container Instances** | Один контейнер без оркестратора | Подойдёт для пилота; проверить SLA и перезапуск. |

**Сеть:** из подсети контейнера/VM должен быть доступ к Slack API, к endpoint’у Databricks workspace (часто публичный или через Private Link — по политике), к Anthropic/LiteLLM и Notion.

**С Databricks на Azure** типичный путь для сервиса — **service principal + OAuth** или **PAT с ротацией**, хранение в Key Vault, выдача в приложение как env при старте. Имеет смысл согласовать с админами workspace: какие scope’ы и политики IAM.

## 5. LiteLLM и Cato в Azure

Запросы к модели идут из процесса бота **по HTTPS** на `LITELLM_BASE_URL` / `ANTHROPIC_BASE_URL` (см. `.env.example`).

**Важно:** Cato на **рабочем ноутбуке** (ZTNA, split tunnel и т.д.) — это **не то же самое**, что сеть **VM / Container Apps / pod в Azure**. У облачного воркера нет «вашего» пользовательского Cato-клиента по умолчанию: исходящий трафик идёт через **подсеть Azure** (NAT, Firewall, политики организации).

| Ситуация | Что обычно делают |
|----------|-------------------|
| LiteLLM доступен с ноутбука «через Cato», с Azure не открывается | Согласовать с **сетью / security**: исходящий доступ с **subnet**, где крутится бот, к хосту LiteLLM (allowlist, статический outbound IP, прокси). |
| Трафик из Azure должен выходить как «корпоративный» | Маршрутизация в **корп. контур** (прокси / FW); при необходимости выставить **`HTTPS_PROXY` / `HTTP_PROXY`** и **`NO_PROXY`** для исключений (например внутренний Databricks, если не должен идти через прокси). Убедиться, что HTTP-клиент в вашем стеке их учитывает. |
| Нужна «сеть через Cato» для облака | Это отдельная интеграция **cloud / site ↔ Cato** (не параметр в `.env`), проект с сетевой командой. |
| LiteLLM только во внутреннем контуре | Внутренний URL / **Private Link** / VPN до той зоны, откуда разрешён доступ; не полагаться на публичный DNS того же имени, что с ноутбука. |

**Практически:** переменные `LITELLM_BASE_URL`, `ANTHROPIC_BASE_URL`, `ANTHROPIC_API_KEY` задают **куда** стучится клиент; **Cato в Azure** — это вопрос **маршрута и политики** с подсети воркера. Имеет смысл уточнить у IT: *«нужен исходящий HTTPS из subnet X в Azure к нашему LiteLLM; с ноутбука ходим через Cato — какой стандарт для cloud workloads (прямой egress, прокси, интеграция с Cato)?»*

## 6. Что сознательно не разбираем в этом файле

- Переход с **Socket Mode** на **Slack HTTP Events** (нужен публичный HTTPS и другой адаптер в коде) — отдельная миграция, если понадобится serverless или строгие ограничения на исходящие WS.
- Детальная настройка Azure AD / Databricks — зависит от вашего tenant; опирайтесь на актуальную документацию Databricks для Azure под ваш тип credentials.
